## 問題

バージョン値（例: `v1.0.0`）が複数のファイルに手書きで重複しており、リリースのたびに全箇所を人手で一致させる必要がある。

**重複箇所:**
- `.github/workflows/sync.yml` の内部 `uses: a24fukuda/issue-to-repository-action@vX.Y.Z`
- `README.md` の利用サンプル（推奨タグ・再利用可能ワークフロー・direct-action の3か所）
- `README.md` のリリース手順の説明
- `.github/workflows/self-sync.yml` のコメント内の説明

GitHub Actions の `uses:` は式（`${{ ... }}`）を書けず静的な文字列しか使えないため、`sync.yml` の内部参照は「これから付けるタグ名」と一致していなければならない。この一致を人手で保つ構造のため取り残しが起きやすく、実際にコミット履歴（`fix leftover @v1 reference` 等）はこの手作業ズレの修正で埋まっていた。整合性を検証する仕組みも存在しなかった。

## 対応方針

`package.json` の `version` を**唯一の真実**とし、2つの仕組みで壊れにくくする（案A+B）。

- **A. CI整合性チェック** — 追跡対象ファイルの全バージョン参照が `package.json` と一致するかを `bun test` で検証し、不一致ならPRを落とす。
- **B. リリース自動化スクリプト** — `bun run release X.Y.Z` で重複箇所を一括書き換え → コミット → 不変タグ作成までを1コマンド化（push のみ手動）。
- 検出・書き換えロジックを共有モジュール（`scripts/version-refs.ts`）に集約し、片方だけ取り残される事故を防ぐ。
- README のリリース手順を新方式に更新し、`self-sync.yml` のコメント内のハードコードされたバージョンを一般化する。

## 実装状況

案A+B を実装済み（PR #7）。加えてセルフコードレビューを2回実施し、検出した所見に対応した。

- **一元化 + 検証:** `package.json` の version を唯一の真実とし、`test/version-consistency.test.ts` が追跡対象の全参照との一致を検証。ズレは PR で落ちる。
- **追跡対象:** `.github/workflows/*.yml`（グロブで自動拡張）+ `README.md`。参照は2種類を追跡 — (1) `uses:` のアクション／ワークフロー参照、(2) ドキュメント中の推奨タグ（バッククォート囲みの `@vX.Y.Z`）。
- **リリース:** `bun run release X.Y.Z` が全箇所を書き換え → コミット → 不変タグ作成。push のみ手動。`--no-git` でファイル編集のみも可。
- **レビュー由来の修正:** 推奨タグのサイレントドリフト、現行版での再実行時の誤エラー、`main()` の未処理 rejection、追跡対象のハードコード、タグ衝突時にファイルを書き換えてから中断していた順序バグ、を修正。裸タグが追跡対象に含まれることを固定するテストも追加。
- CI: 全76テストパス、typecheck 通過、`dist/` 変更なし。

**方針として据え置き（残存リスク／既知の限界）:**
- リリーススクリプトの部分失敗時のロールバックは未実装（git で復旧容易なため低優先）。
- CI が不整合を検出しても「落とすだけ」で自動修正・push はしない（バージョンを上げる判断は人が行う設計）。
- 追跡ファイルのコメント等に無関係な裸タグ `@vX.Y.Z` が入った場合の誤検出リスク。
