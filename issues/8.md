## 問題

バージョン値（例: `v1.0.0`）が複数のファイルに手書きで重複しており、リリースのたびに全箇所を人手で一致させる必要がある。

**重複箇所（当初）:**
- `.github/workflows/sync.yml` の内部 `uses: a24fukuda/issue-to-repository-action@vX.Y.Z`
- `README.md` の利用サンプル（推奨タグ・再利用可能ワークフロー・direct-action の3か所）
- `README.md` のリリース手順の説明
- `.github/workflows/self-sync.yml` のコメント内の説明

GitHub Actions の `uses:` は式（`${{ ... }}`）を書けず静的な文字列しか使えないため、`sync.yml` の内部参照は「これから付けるタグ名」と一致していなければならない。この一致を人手で保つ構造のため取り残しが起きやすく、実際にコミット履歴（`fix leftover @v1 reference` 等）はこの手作業ズレの修正で埋まっていた。整合性を検証する仕組みも存在しなかった。

## 設計上の気づき

当初は「重複する全箇所を一元管理＋自動同期＋CI検証」で対処したが、レビューを重ねる中で**より根本的な観点**に至った。

**バージョンタグと一致していなければならない参照は、実際に実行される `sync.yml` の内部 `uses:` ただ1か所だけ**である。README のサンプルには「このリポジトリの現行版と一致すべき」という正しさの制約が無い（利用者が自分でピン留めするバージョンを選ぶための例であり、古い版を指していても不変タグなので有効に動く）。

つまり、文書を同期対象にしたこと自体が脆さの発生源であり、機械（CI検証・書き換え）はその自己都合の義務を守るために増えていた。**文書から具体版を外せば、脆さの発生源と誤検出リスクが同時に消える。**

## 最終的な対応

`package.json` の `version` を唯一の真実とし、**同期が必要な箇所を実行される1参照に最小化**した。

- **README:** 具体版をすべて `@vX.Y.Z` プレースホルダ＋[Releases](https://github.com/a24fukuda/issue-to-repository-action/releases) リンクに変更。文書は現行版に同期不要となり、陳腐化しない。
- **CI 整合性チェック**（`test/version-consistency.test.ts`, `bun test`）: 追跡対象を `.github/workflows/*.yml`（グロブで自動拡張）に限定し、その内部参照が `package.json` の version と一致するかを検証。不一致は PR で落とす。偽陽性ゼロ。
- **リリーススクリプト**（`bun run release X.Y.Z`）: `package.json` とワークフロー内部参照のみを書き換え → コミット → 不変タグ作成。push のみ手動。`--no-git` でファイル編集のみも可。
- **共有ロジック**（`scripts/version-refs.ts`）: 検出と書き換えで同一パターンを共有し、片方だけ取り残される事故を防ぐ。
- 可変メジャータグ（`v1`）は使わず、不変タグ運用を維持。

レビュー由来の主な修正: 現行版での再実行時の誤エラー、`main()` の未処理 rejection、タグ衝突時にファイルを書き換えてから中断していた順序バグ（前提条件を書き換え前に一括検証）。ドキュメントを追跡対象から外したことで、無関係な裸タグの誤検出リスクも構造的に解消。

**据え置き（設計判断／既知の限界）:**
- リリーススクリプトの部分失敗時のロールバックは未実装（git で復旧容易なため低優先）。
- CI が不整合を検出しても「落とすだけ」で自動修正・push はしない（バージョンを上げる判断は人が行う設計）。
