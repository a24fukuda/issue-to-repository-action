## 背景 / 動機

`v1.0.0` タグは存在するものの、対応する GitHub Release が作成されておらず（`list_releases` が空）、[Releases](https://github.com/a24fukuda/issue-to-repository-action/releases) ページで利用者が「どのバージョンにピン留めできるか」を一覧確認できない状態だった。README の「使い方」も Releases ページを参照するよう案内しているため、Release を整備したい。

現状の `scripts/release.ts` は **コミット + 不変タグ作成** までを自動化し、`git push origin HEAD vX.Y.Z` を唯一の手動ゲートとして残している。GitHub Release オブジェクトの作成は手順に含まれておらず、手動での「Draft a new release」が必要だった。

## 対応方針

タグ push をトリガーにする `release.yml` ワークフローを追加する。

- push は `release.ts` が意図的に手動へ残す唯一の外向き操作であり、その push をそのまま Release 作成のトリガーにできるため、**ローカルに追加の認証情報が不要**（Actions ランナーの `GITHUB_TOKEN` + `contents: write` で作成可能）。
- 「push は手動ゲート」という既存設計と衝突しない（`release.ts` 内に作成処理を入れる案は、Release がタグの remote 存在を前提とするため不採用）。
- ノートは `gh release create --generate-notes` で自動生成し、「ドキュメントを手で同期させない」という repo の方針に合わせる。

## 変更内容

- **`.github/workflows/release.yml`（新規）**
  - トリガー: `on: push: tags: ["v*.*.*"]`（`release.ts` が作る不変タグ形式のみにマッチ。可変メジャータグ `v1` は使わない方針に合致）
  - `permissions: contents: write` のみ（Release 作成に必要な最小権限。既定の read-only トークンでは 403）
  - `gh release create "$tag" --verify-tag --generate-notes` で作成
  - 既存 Release があれば `gh release view` で検出してスキップ → ワークフロー再実行に対して冪等
- **`README.md`**: 「リリース手順」に GitHub Release 自動作成の説明を追記

## 補足 / 既知の制約

- `release.yml` 自体は、実際にタグを切るまでドッグフーディングできない（`self-sync.yml` が `action.yml` を検証するのとは異なり、リリースワークフローは性質上タグ push 時にしか実行されない）。初回のタグ push 時に手動で挙動を確認するのが望ましい。
- 既存の `v1.0.0` は「タグはあるが Release 無し」の状態。この仕組みは今後のタグ push に対して機能するため、`v1.0.0` の Release は手動作成するか、次バージョン（例 `v1.0.1`）を切ったタイミングで自動作成される。
